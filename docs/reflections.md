# 一些感悟

这个项目的主要动机是对John M. Lee的Introduction to Topological Manifolds一书中的定理5.22（每个CW复形是仿紧的）进行形式化验证。原证明占了一页半的纸，本身思路也不甚明了，省略了一些关键的步骤，甚至还包含一处错误。我在第一次阅读的时候花了应该有一周的时间去理解它，甚至到最后我也只能说自己是囫囵的，并不完全确信具体细节。长数学证明的阅读在长期以来一直是我自学数学过程中的挑战。最经典的例子是我学习Walter Rudin的Real And Complex Analysis里面Riesz representation theorem的时候用6页半分10步构造了一个测度并证明其性质。我当时看了得有一个月，每每回过头来想要重新复习，严格证明一些细节，看两页就已经觉得有些吃力，最终只得恭敬地将其放回原处并谨记其表述，不敢妄谈完全掌握。小平邦彦曾讲过：“我觉得，在笔记上反复抄写不懂的数学证明是数学的一个学习方法。如果反复在笔记上抄写仍不明白，那该怎么办才好呢？我也不太知道...”我实在没有做过反复抄写原证明的事情，但我会尝试扩展和改写原先的证明。一旦开始自己写证明，马上就会发现不是写得太详细证明太长就是过分简略到词句的使用近乎晦涩。确实，详略得当结构清晰的自然语言证明需要花费相当的精力。如果你去搜索how to structure long proof,很有可能会被引导到Leslie Lamport的一篇论文：How to Write a Proof.我相当推荐阅读这篇论文的原文，从实际内容和形式来讲都是一篇佳作，而且不需要复杂的理论背景也能看懂。论文主要描述了一种分层（hierarchical）的证明格式。本质上就像写程序，一个证明就像一个函数，内部可能会调用多层子函数，如果你明确子函数的功能且不感兴趣，那就只要看顶层实现就好。机缘巧合我之前了解过一点点lean相关的内容，加之发现论文里推荐的结构比如proof by case和proof by calculation都在lean中有非常好的支持，以及mathlib相对成熟的生态，我决心用lean践行结构化证明。

知易行难。用Lean4+Mathlib完成定理的证明遇到了很多意料之外的困难。一部分原因是自然语言证明过程中本身人会下意识忽略一部分的细节。即使手写Lamport的分层证明，有时也会很难发现，这些细节可能是人所熟知的一些推论（比如一个严格递增的单调序列$a_n$的每一项都满足$n \le a_n$。也可能是某些下意识认为理所应当的几何直观，比如对于n维球$D^{n}$($1 \le n$)内部任意一点p存在一个$f: D^n \to \R$的函数使得f仅在点p为0，在边界为1，内部其余点小于1而外部所有点大于1。忽略掉的这部分内容本身很多时候没有想象力和理解上的困难，但是需要补充非常多的证明细节。另一部分困难来自Lean本身。Lean本身是基于类型论构建的公理系统而不是数学工作者习惯的ZFC公理系统。在ZFC公理体系中如果$S \subset X$，那么S的一个子集自然地也是X的子集（任意两个集合都可以讨论是否存在子集关系），但在Lean中，S的子集和X的子集是两个不同的类型（这与Lean定义集合的方式有关，Lean中集合被定义为类型到Prop类型的函数），不能直接讨论包含关系。再比如在Lean中相当一部分类型是递归定义的，想要处理关于这些类型的命题必须借助其特殊的消去函数(recursor)，而recursor的定义通常比较复杂，使用时需要花费很多精力拼凑出正确的参数类型。我在证明一个等价关系闭包的传递性质的时候分了25种情况进行讨论，虽然大多数情况并不特别复杂。

当然有些时候证明本身的质量不高也是一种困难。在仿紧性的性质证明过程中，教材里面大多数的构造和论证是正确的，但是还是存在疏漏。其定义的bump function $\sigma: D_\gamma^{n+1} \to [0, 1]$的支撑(support)应该位于$D_\gamma^{n+1} \setminus \partial D_\gamma^{n+1}(\varepsilon / 2)$而不是$\partial D_\gamma^{n+1}(\varepsilon / 2)$。递归构造的子复形单位分割以及最终单位分割的支撑闭包位于对应开覆盖中的证明事实上不是平凡的，需要用到其证明的性质(ii)，但在文中只是一笔带过了。这些部分无伤大雅，但形式化验证就是要求每个细节都能禁得起推敲，我们也见过很多因为小的疏漏导致整个证明完全失效的例子。我认为对于作者，尤其是教材作者而言，应该对读者更宽容一些，而对自己更严格一些。宁愿多写点絮叨的“废话”，也不写晦涩的错误。

从CW复形构造定理的末期以及仿紧性定理我才开始使用agent辅助编程。因为我也是从最近才开始在工作中大量使用agent进行编程的。2025年6月我在湾区出差和中学时代的老同学见面聊天，讲到自己的项目，同学说觉得用agent来辅助完成证明的工作应该会更容易些。当时我并没太在意。回想起来，我觉得我对于LLM以及所有形式的人工智能多少还是存在一定傲慢的偏见的。我偏见的根源来自于停机问题：不存在一个判定任意图灵机是否停机的图灵机。程序再厉害，它无法判断任意的程序是否会停止。所以我对于所谓通用人工智能一直持比较冷淡的态度：我认为实际问题规模的扩大速度会大大超过程序增长的速度。当然，这仍然只是推测，不是逻辑论证，所以任何反驳都是可以接受的。神奇的事情在于，前面提到的绝大部分Lean技术细节所带来的困难，可以借助agent以相当高的概率给出可行的解决方案。引入agent绝不是仅仅让我每天能写更多的Lean代码，更重要的是能够让我重新关心更高层次的问题，去理解证明的动机和直觉，而不是湮没在繁杂的细节中。Kline在Mathematical Thought from Ancient to Modern Times里面讨论将自然数的递归构造引入教学的时候谈到，即使其在逻辑上是正确的，它也不是我们熟悉的数字和运算系统了。同样的，逻辑上的正确性是重要的，但不应该是最终目的。这颇有点像青原惟信的三重境界：

+ 看山是山，看水是水：自然语言的数学证明就是证明本身，把完整的证明过一遍就是看完了。

+ 看山不是山，看水不是水：自然语言在阅读时候自动被分解成形式语言，并分析形式语言中各种类型是否能够匹配。证明变得严格了，变长了，但证明产生的动机和直觉被淡化了。

+ 看山还是山，看水还是水：形式语言的具体细节让位于证明的动机和直觉，动机和直觉之外的表述和细节都是自然发生的。

我在大学之前学习数学的时候经常被告知学习数学是“需要天分的”（大学之后我的专业是软件工程，所以没有太多机会听人讲数学了），我个人其实没什么天分，只是比较能够坚持以及乐于接受新的工具以及思想方法而已。我从2024年的11月初读到Lamport的论文开始决定重新学习Lean和mathlib，到2025年的2月中旬开始写第一行关于胞腔复形的定义，再到2026年2月27写完最后一部分关于仿紧性的证明，跨度也超过了一年。离开了学校进入工作，时间也不会特别自由，大部分时间我只能用每天早上早起上班前的一个小时写点，碰上出差那就一行都写不了了。说实话，早起确实挺困的，但能用早起的时间坚持做点自己喜欢的事情，很快乐。
